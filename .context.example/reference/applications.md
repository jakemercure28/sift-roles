# Applications Reference

General playbook for how applications flow through the pipeline.

## Flow

1. **Scraped** — the go-backend orchestrator calls the TypeScript scraper-service, which pulls listings from the ATS boards into the DB.
2. **Scored** — the Go scorer calls Gemini with the resume, context, and JD. Score 1 to 10 lands in the `score` column along with reasoning.
3. **Reviewed** — the dashboard surfaces scored jobs. You decide approve / reject.
4. **Applied** — applying is always manual. Use `/app-questions` to draft answers in your voice, then fill and submit the form yourself. The app never submits applications and never marks a job as applied on its own.
5. **Tracked** — the events table records stage changes. Rejection likelihood is re-scored when you mark a job applied.

## Scoring rubric

The scorer prompt asks for a 1 to 10 score along these dimensions:

- Stack match (do the technologies line up?)
- Seniority fit (is this the right level?)
- Compensation signals (if the JD includes a band, does it meet the floor?)
- Company stage (is it a known fit?)
- Overall desirability (given everything else)

The full prompt lives in the Go scorer (`internal/scorer`). Tweak it per profile rather than per applicant, so it remains consistent across runs.

## Form questions

Applying is manual. When a form asks custom questions, use `/app-questions`: paste the questions (or an Ashby job URL/ID) and the LLM drafts answers grounded in the applicant's resume, context, and voice rules. You review, fill, and submit the form yourself.

## Voice check

Any answer generated by the LLM is run through the go-backend `voice-check` subcommand (`internal/voice`) which flags em dashes, buzzwords from the kill list, and suspiciously AI-flavored sentences. Flags are shown in the dashboard; you can edit in place or click "rewrite" to ask the LLM to try again with the issues highlighted.
